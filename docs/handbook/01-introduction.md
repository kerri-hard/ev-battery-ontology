## 1.1 한 문장 요약

> **EV 배터리팩 제조 공정을, 사람이 24시간 지키지 않아도 AI가 스스로 문제를 감지·진단·복구·학습하는 "다크 팩토리(dark factory)"로 만드는 플랫폼.**

핵심은 *대시보드*가 아니라 **자율 복구 엔진**입니다. 운영자 화면은 AI가 무엇을 하고 있는지 *감독*하기 위한 워크벤치이지, 사람이 직접 조작하기 위한 콘솔이 아닙니다.

---

## 1.2 왜 이 문제인가

EV 배터리 제조의 현실:

- **공정 복잡도** — 5 공정영역(셀 어셈블리 → 전장 → 냉각 → 인클로저 → 최종 검사) × 31 스텝
- **수율 손실 = 수십억 원/년** — 표면 처리 코팅 0.988 수율, 와이어 하네스 0.98 수율 같은 작은 단위가 누적
- **장애 데이터의 휘발성** — 트러블슈팅이 운영자 머릿속에만 남아 신규 인력 학습 불가
- **24시간 운영 부담** — 야간 교대조 인력 확보, 휴먼 에러

해결 가설: **공정 지식 = 온톨로지 그래프**, **반복 트러블슈팅 = 에이전트 자동화**, **인과관계 = L3 그래프 추론**, **신규 패턴 = LLM 보강**.

---

## 1.3 5단계 자율성 로드맵 ([VISION.md](../../VISION.md) §6)

| 단계 | 이름 | 사람의 역할 | 시스템의 역할 |
|---|---|---|---|
| **L0** | 수동 | 모든 결정 | 모니터링만 |
| **L1** | 권고 | 결정 | 권고안 제시 |
| **L2** | 반자율 | HITL 승인 | 액션 제안 + 실행 |
| **L3** | 자동 (현재 단계) | 예외 검토 | 99% 자동 복구 |
| **L4** | 완전 자율 | 거버넌스 | 전권 + 자가 진화 |

**현재 단계**: **L3 자율 공장 + L4 Tier 2** — 자동 인과 발견·자가 진화·LLM 오케스트레이션·CDT 시뮬레이션·Multi-step PRE-VERIFY·Counterfactual replay·HITL inline·Sim Control 모두 구현됨. **L4의 마지막 1마일** = 사람이 정책만 정의하고 시스템이 모든 운영을 자동화하는 단계가 다음 목표.

## 1.4 9가지 설계 철학 ([VISION.md](../../VISION.md) §9)

본 프로젝트의 코드 결정은 모두 다음 9 원칙 하나 이상에 근거를 둡니다.

| # | 이름 | 한 줄 |
|---|---|---|
| 9.1 | **온톨로지 = 두뇌** | 모든 지식을 그래프로 구조화. 코드 if-else 금지 |
| 9.2 | **인과 추적** | 상관관계가 아니라 인과를 명시 (CausalRule, FailureChain) |
| 9.3 | **Hybrid Agentic** | 빠른 규칙 에이전트 + 복잡 추론 LLM |
| 9.4 | **자동 복구가 목적** | 대시보드가 아니라 "사람 없이 해결" |
| 9.5 | **실패에서 성장** | 장애·오탐을 삭제 안 하고 학습 데이터로 축적 |
| 9.6 | **점진적 자율성** | L1 → L2 → L3 → L4 순서 승격 |
| 9.7 | **에이전트 협업** | 단일 책임. 한 에이전트 = 한 역할 |
| 9.8 | **온톨로지 쿼리 우선** | 판단은 그래프 쿼리에 기반. 하드코딩 금지 |
| 9.9 | **스키마 idempotent 확장** | `try ... except`로 재실행 안전 |

## 1.5 EV 배터리팩 도메인 컨텍스트

본 시스템이 다루는 가상 공장의 제품:

- **96셀 파우치형 리튬이온 배터리팩** (전기차 1대 분량)
- **5 공정영역**:
  - **PA-100 셀 어셈블리** — 셀 검수, 스택, 탭 용접, EOL 시험
  - **PA-200 전장 조립** — BMS, 와이어 하네스, 커넥터
  - **PA-300 냉각 시스템** — 쿨링 플레이트, 누설 검사
  - **PA-400 인클로저** — 하우징 가공, 표면 코팅, 가스켓
  - **PA-500 최종 조립/검사** — 통합 시험, 에이징, 출하 검사
- **31 공정 스텝** — 위 5 영역 안에서 세분화됨 (Ch.04 참조)
- **주요 병목** — PS-203 와이어 하네스(수율 0.98), PS-404 표면 코팅(수율 0.988)
- **핵심 자재** — 리튬이온 셀(SK On 가상), BMS PCB(LG이노텍 가상), 쿨링 플레이트(삼성SDI 가상)
- **안전 등급** — A(고전압/레이저, 위험), B(주의), C(일반). PRE-VERIFY 임계가 등급별로 다름

> 도메인 데이터는 모두 [`data/graph_data.json`](../../data/graph_data.json)에 정의. 하드코딩이 아니라 데이터 주도.

## 1.6 두 가지 하네스 — v3와 v4

본 저장소에는 두 개의 메인 하네스가 공존합니다.

| 하네스 | 목적 | 위치 | 트리거 |
|---|---|---|---|
| **v3 (`src/v3/`)** | 멀티 에이전트 토론으로 *온톨로지 자체*를 개선 | `src/harness_v3.py` | `./run.sh v3` |
| **v4 (`src/v4/`)** | 7-페이즈 자율 복구 + L3-L4 거버넌스 | `src/v4/engine.py` | `./run.sh dev` |

v3는 도메인 지식 빌더(graph가 잘 자라는가?), v4는 운영(graph로 incident가 자동 복구되는가?). 둘 다 같은 Kuzu DB를 공유하며, v3가 만든 인과 규칙을 v4가 판단에 사용합니다.

## 1.7 무엇을 *안* 만드는가

명시적으로 범위 밖인 것:

- 🔴 **물리 PLC 직접 제어** — 본 시스템은 OPC-UA/Sparkplug B 인터페이스만 제공. 실제 액추에이터는 별도 시스템.
- 🔴 **사람 대체 의사결정** — A 등급(고위험) 액션은 항상 HITL.
- 🔴 **블랙박스 학습 모델** — 모든 추론은 추적 가능해야 함. `LearningRecord` 영속화 필수.
- 🟡 **타사 ERP/MES 통합** — 향후 로드맵. 현재는 시뮬레이션 데이터로 자율 루프 검증.

## 1.8 핵심 약속

이 책은 다음을 약속합니다:

1. **모든 주장은 코드 경로와 같이 제시** — 추상 개념 옆에 항상 `src/v4/foo.py:bar` 형태 인용
2. **용어는 Ch.10 글로서리 1회 정의** — 챕터 안에서 약어/한글 혼용
3. **예제는 실제 동작 코드** — 의사 코드 사용 시 명시
4. **갱신 정책 명시** — 코드와 책이 어긋나면 코드가 정답

다음 챕터에서 5분 안에 첫 incident 자동 복구를 직접 보겠습니다.
